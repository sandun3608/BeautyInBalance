Introduction  
This  is  the  API  reference  for  Koko's  APIs.  The  checkout  API  is  used  to  create  a  payment  request  for  
a
 
Koko
 
user.
  This  is  a  developer  preview  of  Koko's  Platforms  APIs.  This  documentation  will  be  updated  as  we  
introduce
 
new
 
features.   
How  it  works  
The  following  diagram  illustrates  how  a  payment  ﬂow  works  between  the  Koko  and  the  
merchant’s
 
eCommerce
 
site:
      
Authentication  
You  will  receive  the  following,  ●  Merchant  ID  ●  API  Key  ●  RSA  Public  Key  for  the  Koko  environment  ●  RSA  Private  Key  for  merchant-speciﬁc  API  signing   Note:  The  API  credentials  are  separate  from  your  Merchant  ID,  and  it  is  possible  to  have  several  
API
 
credentials
 
associated
 
with
 
the
 
same
 
Merchant
 
ID
 
but
 
different
 
Cashier
 
accounts.
  The  credentials  should  be  sent  as  an  authorization  header  for  every  request  with  the  username  
and
 
password.
  You  receive  both  your  live  and  test  credentials  by  contacting  the  Koko  Merchant  Success  team  for  
your
 
country.
 If  the  credentials  are  missing  or  wrong,  Koko  will  respond  with  401  Unauthorized .    Note:  Below  endpoints  examples  are  shown  using  production  urls,  Please  found  relevant  QA  and  
Dev
 
endpoints
 Dev :  devapi.paykoko.com  QA :  qaapi.paykoko.com  PROD :  prodapi.paykoko.com

URL

the
 
mentioned

using
 
the
 
_returnUrl
 
set

the
 
createOrder
 
api
 
is
 
called.
  eg:  http://eComm.com/orders/66/returned/3    Parameter  Example  Description

trnId  f7ec3q588eq435zfc8c396d3ecd3306a

Response  Url  (_responseUrl)   If  the  payment  is  successful,  Koko  backend  will  send  an  api  request  to  the  url  specified  in  
‘_responseUrl’
 
parameter.
 
Example
 
given
 
below.
  It  will  be  a  HTTP  POST  api  call.  Using  the  ‘_responseUrl’  parameter.  Content  type  will  be  
"application/x-www-form-urlencoded".
  Therefore  it  is  the  eCommerce  responsibility  to  set  up  an  endpoint  supporting  this  for  the  url  
given
 
in
 
‘_responseUrl’.
 
Otherwise
 
eCommerce
 
will
 
not
 
have
 
any
 
way
 
to
 
verify
 
if
 
the
 
payment
 
has
 
been
 
successfully
 
done
 
or
 
not
 
from
 
the
 
backend.
 
Else
 
eCommerce
 
will
 
have
 
to
 
rely
 
on
 
front
 
end
 
redirection
 
only,
 
which
 
is
 
not
 
100%
 
secure.
